Confidential 10 / 41
Modular Monolith · Domain-based SOA

The pragmatic middle ground

Within each PBC we apply a modular-monolith design: clear domain modules with enforced boundaries, deployed as a single, efficient unit. This gives us microservice-grade separation of concerns without distributed-systems overhead — and the freedom to extract a module into its own service when scale demands it.

  • Domain modules map directly to commerce processes
  • Explicit contracts between modules, no leaky coupling
  • Single deployable — simpler ops, lower latency
  • Extract-on-demand — promote a module to a service when justified

Rule · Evaluation · Action engines

Business logic that ops teams control

Rule EngineMicrosoft Rules Engine
→
Evaluation Enginescore, best-choice…
→
Action Engineexecute outcome

A super-flexible rule engine — built on the Microsoft Rules Engine — expresses any complex scenario; the evaluation engine scores options with a best-choice (and similar) mechanism; the action engine executes the result. The same rule · evaluation · action infrastructure is reused everywhere across the platform — sourcing, campaigns, payments, dispatch and content.

  • Sourcing — score-based allocation by cost, satisfaction, capacity
  • Campaigns — priority groups & best-choice selection
  • Payments & dispatch — rule-based routing & scoring

Toggles & user exits

Change behaviour without changing the core

Toggle-based structure

Feature flags switch capabilities on/off per tenant, environment or rollout — enabling safe, gradual releases and A/B experiments without redeployment.

User exits — DB & code level

Defined extension points let clients inject custom logic at both the database layer (stored procedures) and the application layer — tailoring behaviour while staying on the upgrade path.

The result: deep per-client customisation that never forks the shared codebase — so every tenant keeps receiving platform upgrades.

Network architecture

Segmented, isolated, zero-trust

VPC & SUBNET LAYOUT
  • Default VPC — basic shared services
  • Dedicated production VPC — hardened, separated services
  • Private container / database / general subnets — fully closed to the internet
  • Public subnet — VPN & FTP only, IP-restricted + MFA
ACCESS CONTROL
  • Per-application security groups with least-privilege rules
  • Egress via NAT Gateway only — no direct outbound
  • Zero-trust network — VPC peering & private links
  • AWS Security Hub — continuous audit & auto-remediation

Each component sits in its own segment, shrinking the attack surface and maximising isolation.
